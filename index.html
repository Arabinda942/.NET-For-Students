Print the message "I Love SMU" to the console.

Instructions

  • Use Console.WriteLine() to print the message.
Main.cseditable
Click "Compile & Run" to see output here.

What it does: Console.WriteLine is the simplest way to send text to standard output, adding a newline afterwards. There's no input, no variables — it exists purely to prove the compile-run-print loop works end to end, which is why it's worth 10 marks despite being one line: it's the checkpoint that your environment, syntax, and entry point (Main) are all correctly wired.

Hidden theory: the anatomy of a C# program

using System; imports the System namespace, which is where Console, basic types, and math helpers live. Without it you'd have to fully qualify everything as System.Console.WriteLine.

Every executable C# program needs exactly one entry point: a static void Main(string[] args) method inside a class. static means it belongs to the class itself, not to any instance of it — the runtime calls it before any object exists. string[] args carries command-line arguments, which you can ignore if you don't need them.

Console.Write vs Console.WriteLine: the first leaves the cursor on the same line, the second appends \n (or \r\n on Windows). This distinction matters a lot once you start building prompts like "Enter a number: " followed by input on the same visual line.

Click "Compile & Run" to see output here.

What it does: Console.ReadLine() always returns a string — even if the user typed digits. Convert.ToInt32(...) parses that string into an int so it can be used in arithmetic; without it, a + b would just glue two strings together ("12" + "30" = "1230") instead of adding numbers.

Hidden theory: type conversion in C#

C# is statically typed — every variable's type is fixed at compile time. Converting between types is explicit rather than automatic, which is why Convert.ToInt32 (or int.Parse) is required instead of just assigning a string into an int.

Convert.ToInt32 throws a FormatException if the string isn't a valid whole number, and an OverflowException if it's outside the int range (roughly ±2.1 billion). int.TryParse is the safer alternative when you don't want an exception for bad input — it returns true/false instead.

Note the difference between Convert.ToInt32(null) (returns 0) and int.Parse(null) (throws ArgumentNullException) — a subtle but common interview question.

What it does: each branch asks "am I bigger than both of the others?" — that double condition is what makes the logic correct with only three comparisons instead of needing every possible pairwise check. If the first two conditions fail, the third number must be the largest by elimination, so the final else needs no condition at all.

Hidden theory: short-circuit evaluation

&& and || in C# are short-circuiting: in num1 >= num2 && num1 >= num3, if the first comparison is already false, the second is never evaluated. This matters more once conditions have side effects or could throw (e.g. arr.Length > 0 && arr[0] == 5 safely avoids an index error on an empty array).

if / else if / else chains are evaluated top to bottom, and only the first matching branch runs — order can matter when ranges overlap.

Hidden theory: for-loops and off-by-one errors

A for loop has three parts: init; condition; increment. Here int i = 1 starts the counter, i <= 10 keeps it going while true, and i++ increases it after each pass. Using i < 10 instead would stop one iteration early (a classic off-by-one bug) and give 45 instead of 55.

Automated grading typically checks exact string equality against an expected value, not just "did the numbers work out" — which is why a spelling mistake in an otherwise-correct program can still fail every test case, as it does here.

  • Print words.Length.
Main.cseditable
Custom input (a single sentence)
Click "Compile & Run" to see output here.

What it does: string.Split(' ', StringSplitOptions.RemoveEmptyEntries) breaks the sentence into an array wherever a space appears, and the RemoveEmptyEntries option throws away any empty strings that would otherwise appear from double spaces or leading/trailing whitespace. words.Length — the array's element count — is then the word count.

Hidden theory: strings and arrays in C#

A C# string is an immutable sequence of characters — every operation like Split, Trim, or ToUpper returns a new string or array rather than modifying the original in place.

Split without RemoveEmptyEntries would count consecutive spaces as producing empty "" elements, silently inflating the word count on messy input — a common source of subtle bugs in text-processing exercises.

Arrays in C# have a fixed size once created and expose it via .Length (not .Count(), which is a LINQ extension method for any sequence, including arrays, lists, and more).

Hidden theory: exception handling in depth

All C# exceptions derive from System.Exception. FormatException and the more general Exception both fit that hierarchy, which is why catch (Exception ex) would technically also catch a FormatException if it appeared first — catch order matters: put the most specific exception types before the more general ones, or the general block "steals" cases meant for a more specific handler.

throw new Exception("message") creates and raises an exception object on demand — useful for enforcing business rules (like a valid numeric range) that the language itself has no built-in check for.

finally runs whether the try succeeded, an exception was caught, or even if an exception was not caught at all — it's the right place for cleanup code (closing files, releasing connections) that must happen regardless of outcome.

Overusing exceptions for ordinary control flow (like validating input, which happens on every run) is considered less idiomatic in C# than int.TryParse style validation — this exercise deliberately uses exceptions to teach the mechanics.

What it does: Animal is the base class holding a shared Name field and an Eat() method. Dog : Animal is the derived class — the colon means "Dog inherits from Animal" — so every Dog automatically has Name and Eat() for free, and adds its own Bark(). : base(name) in the constructor forwards the name up to the parent's constructor so it gets initialised in one place.

Hidden theory: inheritance and the "is-a" relationship

Inheritance models an is-a relationship: a Dog is an Animal, so it makes sense for it to reuse Animal's behaviour rather than duplicating it. Contrast this with composition ("has-a"), where a class holds a reference to another object instead of extending it — often a more flexible design in practice.

C# supports only single inheritance for classes (one direct base class), unlike some languages that allow multiple. Shared behaviour across unrelated classes is instead achieved through interfaces.

public access lets a member be seen from outside the class; marking Name as protected instead would let derived classes use it while still hiding it from the outside world — generally the safer default for inherited state.

What it does: public static Number operator +(Number a, Number b) teaches the compiler what n1 + n2 means when both sides are Number objects — instead of adding the objects themselves (which is meaningless), it returns a new Number wrapping the sum of their Value fields.

Hidden theory: operator overloading

Overloadable operators in C# include arithmetic (+ - * / %), comparison (== != < >), and a few others — but not all of them (e.g. && and || can't be overloaded directly). Overloads must be public static because they're not tied to a single instance; they take both operands as parameters.

Overloading == conventionally means you should also override Equals() and GetHashCode(), so the type behaves consistently in collections like Dictionary and HashSet.

Used well, operator overloading makes domain types (money, vectors, complex numbers) read naturally in code; used poorly, it can make + do something surprising — a common design-review concern.

Main.cseditable · needs a real compiler

What it does: Student holds the name behind a protected field — visible to Student itself and anything that inherits from it, but not from outside. Marks : Student inherits that name-handling for free and layers on its own private marks field and matching setter/getter-style methods, so one Marks object can show both pieces of data.

Hidden theory: access modifiers

public — visible everywhere. private — visible only inside the declaring class. protected — visible inside the declaring class and any class that derives from it, which is exactly why Marks can rely on name being set correctly without redeclaring it.

This is encapsulation in practice: each class exposes only the methods needed to interact with it (SetName, ShowName) while keeping its internal storage details private or protected — callers don't need to know or care how the name is stored.

  • Create a new blank file and confirm with messages.
Main.cseditable · needs a real compiler + filesystem
This program touches the real filesystem, so it can only run on an actual machine (Visual Studio, dotnet run, etc). Click above to see the worked example output.

What it does: File.Exists(path) checks the disk before doing anything destructive. If a file is already there it's deleted and the deletion is confirmed; either way, File.Create(path).Close() makes a fresh empty file — .Close() matters because File.Create returns an open file stream, and leaving it open would lock the file.

Hidden theory: file handling and resource cleanup

System.IO is the namespace for file and stream operations: File (static helpers for whole-file operations), FileStream, StreamReader/StreamWriter for text.

File handles are an unmanaged resource — the garbage collector doesn't reliably clean them up promptly, so they must be closed explicitly (or wrapped in a using statement, which calls Dispose()/Close() automatically even if an exception occurs). Forgetting this is a classic cause of "file is being used by another process" errors.

Idiomatically, this whole method is safer written as using (File.Create(path)) { }, which guarantees the stream closes even if something throws in between.

Theory Vault

The concepts underneath the ten projects above, gathered in one place for quick review before a quiz or interview.

The four pillars of OOP

Encapsulation — bundling data and the methods that operate on it, hiding internal state behind controlled access (private fields, public methods/properties).

Abstraction — exposing only what's necessary through a simplified interface (e.g. an abstract class or interface) while hiding implementation detail.

Inheritance — a class acquiring the members of a base class, modelling "is-a" relationships and enabling code reuse.

Polymorphism — the same method call behaving differently depending on the actual runtime type of the object, via overriding (runtime) or overloading (compile-time).

Value types vs reference types

int, double, bool, char, and structs are value types — assigning one variable to another copies the value. Classes, arrays, and strings are reference types — assigning copies the reference, so both variables point at the same underlying object.

This is why passing a large custom class into a method and mutating a field inside it affects the caller's object too, while passing an int and changing it inside the method does not affect the caller's copy — unless you use ref.

Compilation model: from .cs to running program

The C# compiler (csc or the one built into dotnet build) turns your .cs source into IL (Intermediate Language) bytecode inside a .dll/.exe. The .NET runtime (CLR) then JIT-compiles that IL into native machine code at execution time — this is what makes .NET code both portable across platforms and fast to run.

Exception hierarchy at a glance

System.Exception is the root. Common built-ins beneath it include SystemException → FormatException, OverflowException, NullReferenceException, IndexOutOfRangeException, DivideByZeroException, and ArgumentException (with ArgumentNullException and ArgumentOutOfRangeException beneath that). Custom exceptions should derive from Exception and typically add a constructor that forwards a message.

Why tests fail even when the "logic" is right

Two of the projects above (04 and 06) demonstrate this directly: automated graders usually assert exact string equality against an expected output. A correct algorithm wrapped in a misspelled or incomplete message still fails every test case, because the grader has no notion of "close enough" — it's comparing characters, not intent. Always re-read the exact expected string in a problem statement before assuming a failing test means your logic is wrong.
